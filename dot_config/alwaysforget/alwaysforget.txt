# SSH
## Order Precedence
SSH takes the first specified value for an option.

That is if multiple Host groups specify the same option, the first one takes precedence, with
some exceptions (like IdentityFile, which appends to the files SSH will try)

If an option is specified outside of a Host group, then it will always be evaluated,
but it won't necessarily be applied. Some examples

```ssh
ServerAliveInterval 30

Host staging-web1
    User deploy
Host staging-*
    User admin
    IdentityFile file1
Host *
    User default
```

A connection to `staging-web1` uses `deploy`, not `admin` or `default`.

SSH processes top to bottom, one line at time.

Host <patterns> are like `if` statements. They are not `else if`s.

SSH goes through Host <pattern> patterns in order and applies as many options as possible.
They fall through so to speak.

In the above example, although the `deploy` user takes precedence, SSH will still
apply the `IdentityFile thing` option specified in the second Host pattern block.

If we were to add `ServerAliveInterval 50` to `Host *`, it would have no affect,
since the `ServerAliveInterval 30` at the top of the file would take precdence.

If we were to move `ServerAliveInterval 30` to the bottom of the SSH file, and then
add `ServerAliveInterval50` to `Host *`, `ServerAliveInterval 50` would take precedence

translating that ssh config to python

```python
def update(option, value):
    if option == None:
        return value
    return option

# host is supplied as an argument to this program
# options is the SSH options SSH will use after evaluating this config file
options.ServerAliveInterval = update(options.ServerAliveInterval, 30)
if host == 'staging':
    options.User = update(options.User, 'deploy')
if host.startswith('staging-'):
    options.User = update(options.User, 'admin')
    options.IdentityFilesToTry.append('file')
# always true
if host is not None:
    options.User = update(options.User, 'default')
```


## Match
Match blocks lets you match on things other than just Host
They are applied in order just like Host blocks

## ControlMaster
If you have ControlMaster=auto then your ssh session will be kept alive
making any changes you make to your ssh config that affect that host null until the
controlmaster socket is removed

To be safe remove all ControlMaster sockets before re-testing

## Commandline flags
For OpenSSH and Tectia clients, the order of precedence is, from strongest to weakest:

1. Command-line options
2. The user’s local client configuration file
3. The global client configuration file

This goes for even if you specify -F <file>

If you look at the example ssh file above again and then specified
ssh -o User=other -F ssh-config staging-web1
It would use ssh as the `other` user 

Same is true for
ssh -F ssh-config -o User=other staging-web1

# DOCKER

## CMD VS ENTRYPOINT
 
FOR CMD

docker run image
will run CMD as the default

docker run image otherthing

will run otherthing instead of CMD

FOR ENTRYPOINT

docker run image
will run ENTRYPOINT as the default

docker run image otherthing
will run ENTRYPOINT otherthing

## Docker networking

none:
    no networking, container is completely network isolated

bridge:
    docker makes a bridge interface that has a subnet
    subnet gets nat'd to hosts ip address for ipv4
    for ipv6 they also get nat'd to the hosts single ipv6 address
    individual ports need to be forwarded from host to containers

    proxmox also supports NAT using similar iptables rules that docker bridge adds
    https://pve.proxmox.com/wiki/Network_Configuration#sysadmin_network_masquerading

    `ip a` will print an address that is NAT'd to the hosts IP.
    for ipv4 this is double nat, which is sad. for ipv6 it's just one nat

host:
    container uses hosts networking stack. uses the same ip address as the host
    no port forwarding required
    `ip a` will print the same thing you see on the host

macvlan:
    each container gets an interface with its own mac address
    uses promiscuous mode of network card to send frames with different mac address
    no bridges are created
    sketchy as fuck

    at work so can't check if it gets a real ip address from the lan or not via dhcp over wifi
    they have WPA3 enabled which the container doesn't know how to do

    to have a container on a vlan, you make a vlan interface on the parent interface
    then assign the container to a macvlan network pointed at that vlan interface

ipvlan:
    two modes

    l2:
        similar to macvlan, no bridge is created
        each container has its own IP, but uses the host interface's mac address

    l3:
        ip forwarding on the host interface basically. host acts as a router
        each container gets assigned an ip as part of a subnet
        subnets are routed from host's interface (on its own subnet) to the container's subnet
        drops all broadcast and multicast traffic, no bridging loops
        containers that use the same host routing interface can ping each other

        needs to be on separate subnet as the default namespace since it requires a netlink
            route in the default namespace pointing to the ipvlan parent interface

        other hosts on the network need to have routes pointing to the docker host
        in order to reach containers in the ipvlan l3 mode networks

# SYSTEMD DON'T CUT OFF MY OUTPUT
When serial'd into a device systemd won't wrap lines by default
You can make it wrap lines with these commands

SYSTEMD_LESS="FRXMK" journalctl -b
or just
journalctl -b | cat
